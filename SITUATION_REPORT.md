# Nippo System: Project Status Report for AI Consultation

このドキュメントは、外部 AI が本プロジェクトの現状・アーキテクチャ・課題を即座に理解し、高度な技術コンサルティングを行うためのコンテキストを提供します。

## 1. プロジェクトの目的
PC上のあらゆる活動（画面、入力、音声等）を記録し、その生データから「意味のある文脈」を抽出。最終的に LLM（大規模言語モデル）を用いて、極めて高精度で洞察に富んだ**自動業務報告（日報）**を生成することを目的としています。

## 2. 現在のアーキテクチャ（4段階パイプライン）

### Stage 1: Raw Data Collection (収集形態)
- **OCR ユニット**: 二画面対応の高精度キャプチャ ＋ ローカル Tesseract OCR による TSV ストリーム生成。差分検知による負荷軽減を実装済み。
- **Input Monitor**: マウス（座標・クリック）およびキーボード（生キー）を別プロセスで常時監視し、TSV に記録。

### Stage 2: Semantic Transformation (意味付け)
- **main_transformer.py**: 生データを「意味のある断片」に変換。
  - **Click Mapping**: 画面座標を OCR 文字列にヒットさせ、「どのボタンを押したか」を特定。
  - **Key Logic**: 生の打鍵から「清書された文章」を復元。Ctrl+C/V 等を `[COPY]`, `[PASTE]` などのラベルに置換し、編集強度（Edit Intensity）を算出。
  - **Context Resolution**: ウィンドウタイトルから業務カテゴリ（例：開発、調査、通知）を判定。

### Stage 3: Narrative Aggregation (物語化)
- **aggregator.py**: 全イベントを時系列で統合し、YAML 形式の「Work Story」を生成。
  - **Idle Detection**: 5分以上の無操作時間を `[IDLE]` としてマーク。
  - **Active Density**: 各作業セッションの集中度（1分あたりの操作数）を算出。

### Stage 4: AI Prompt Generation (アウトプット)
- **ai_prompts/**: 生成された YAML データを LLM に流し込むためのテンプレートと README。現在は手動でのコピペ投稿を主眼に置く（セキュリティ・柔軟性重視）。

## 3. OCR ユニットの出力仕様 (Raw OCR Output Specification)
Stage 1-2 で生成される、加工前のセマンティック・ストリーム（TSV）の形式です。外部AIへの入力ドラフトとして検討可能です。

### Output: `ocr_stream_YYYY-MM-DD.tsv`
- **形式**: タブ区切りテキスト（TSV）
- **カラム**: `[Time]\t[BlockID]\t[Normalized_Box]\t[Text]`
- **Normalized_Box**: `[ymin, xmin, ymax, xmax]` (0-1000スケール)
- **例**:
  ```text
  03:15:20	12	[120, 450, 145, 600]	Submit Button
  03:15:25	15	[800, 20, 850, 300]	Console Log: Success
  ```

## 4. 現在の性能・精度指標 (Performance & Metrics)

### 処理速度 (Processing Speed)
- **OCR処理時間**: 1フレームあたり **1.5 〜 2.5秒** (AMD Ryzen 7環境)
- **キャプチャ間隔**: 5.0秒（デフォルト）
- **遅延**: 非同期処理により操作自体は妨げないが、解析にバックログが発生する場合がある。

### 過去の精度実験データ (Historical Accuracy Experiments)
`ocr_lab` 環境での、フォントサイズと認識一致率（一致率）の相関データです。
**Smart Scaler (Target Height 45px)** の導入により、微小文字の精度を劇的に改善しました。

| フォントサイズ (px) | 補正なし (Static 1.0x) | **Smart Scaler (45px化)** | 備考 |

## 5. 最近の主要な技術的成果
- **時間的マッチングの解決**: クリック時刻に対し、直近3つの OCR フレームを遡って検索することで、画面遷移時のラグによるマッチング漏れを極小化。
- **座標正規化**: OCR の 0-1000 スケールとマウスのピクセル座標を正規化して統合。
- **生産性指標の導入**: 単なる記録ではなく、「アイドル時間」や「活動密度」をデータに落とし込むことで、思考時間の可視化を実現。

## 6. 現在の課題と今後のビジョン（開発者の想い）

### 技術的な課題
- **ノイズの低減**: フォーカス移動のための背景クリックが多いため、更なるフィルタリングや領域別の重要度判定が必要。
- **OCRのチューニング**: ユーザーや環境によって最適な OCR エンジン（より高速、より精度重視等）を切り替えられるようにしたい。
- **マルチウィンドウ解析**: アクティブでないウィンドウ（裏の Slack 等）の変化もコンテキストに含めたい。

### アプリケーション化・GUI
- **可視化**: 現在のドキュメントベースの管理から、ユーザーのデスクトップに常駐する GUI アプリへの移行を検討中。
- **ON/OFF トグル**: 各機能の稼働状況やエラーの有無を視覚的に把握でき、ワンクリックで YAML 出力・日付選択ができるインターフェース。

### 商業的・実用的な視点
- **差別化**: 既存の「ただの記録」ツールではなく、最新の OCR/AI 技術をどう組み合わせて「売り」にするか。
- **API vs 手動**: 法人向けには API 連携が必要だが、個人利用ではセキュリティ上、手動投稿の方が安心できる。

### OCR Engine Comparison: Tesseract vs. Windows.Media.Ocr
外部AIとの相談用に、エンジン選定の背景を整理しました。
- **Tesseract (現状)**: 
  - メリット: **確信度 (Confidence)** が取得可能。UI特有のノイズ（枠線等）に対する「Adaptive Scaling（適応スケーリング）」や「Morphological Tuning」などの独自チューニングが可能。
- **Windows.Media.Ocr (検討中)**: 
  - メリット: **極めて高速** (ハードウェア加速)。OS標準で安定。
  - デメリット: 標準APIでは**確信度が取れない**。

1. **「思考時間」の定義**: 無操作時間は「思考中」なのか「放置」なのか。これをデータ構造的にどう区別すべきか？

3. **プロダクトの絞り込み**: 最小構成（MVP）としてどこにフォーカスすべきか？
4. **性能負荷**: 処理が他の業務を邪魔しないよう、さらなる最適化（特にOCR周り）の戦略は？
